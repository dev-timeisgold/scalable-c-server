# C 네트워크 서버 프로젝트 (기본부터 대용량 처리까지)

C 언어의 소켓 프로그래밍을 이용해 기본 TCP 에코 서버를 구축하고, 발생하는 구조적 문제점을 찾아내며 점진적으로 성능을 개선해 나가는 시스템 아키텍처 학습 프로젝트입니다.

## 📌 진행 단계
- [x] Phase 1: 기본 TCP 에코 서버 (단일 스레드 / Blocking I/O)
- [x] Phase 2: 멀티 스레드 서버 (접속자당 1개의 스레드 할당)
- [x] Phase 3: 멀티플렉싱 서버 (epoll을 이용한 이벤트 기반 단일 스레드 처리)

---

## 🛠 Phase 1: 기본 TCP 에코 서버 (단일 스레드)

### 구현 내용
리눅스 네트워크 기본 시스템 콜을 순서대로 사용하여 통신 흐름을 구축했습니다.
1. `socket()`: TCP 통신을 위한 소켓(FD) 생성
2. `setsockopt()`: 서버 비정상 종료 시 포트가 잠기는 에러 방지 (`SO_REUSEADDR`)
3. `bind()`: 소켓에 포트 번호(8080) 지정
4. `listen()`: 클라이언트 접속 대기열(Queue) 생성
5. `accept()`: 대기열에서 클라이언트를 꺼내와 통신할 새로운 소켓 생성
6. `read()` / `send()`: 클라이언트의 메시지를 수신하고 그대로 다시 전송(Echo)
7. `close()`: 통신 종료 후 리소스 반환

### 아키텍처의 한계 (Phase 2 도입 배경)
단일 스레드 구조이기 때문에 발생하는 **블로킹(멈춤) 문제**가 존재합니다. 
한 클라이언트가 접속한 후 메시지를 보내지 않고 대기하면, 서버가 `read()` 함수에서 블로킹 상태에 빠집니다. 이로 인해 대기열에 있는 다른 모든 클라이언트가 서버에 진입조차 하지 못하는 치명적인 병목이 발생했습니다.

## 🛠 Phase 2: 멀티 스레드 서버

### 구조 변경 내용
Phase 1의 블로킹 문제를 해결하기 위해, 메인 스레드와 워커 스레드의 역할을 완벽히 분리했습니다.
* **메인 스레드:** `accept()`를 통해 클라이언트의 접속만 수락하고 다시 대기 상태로 돌아갑니다.
* **워커 스레드:** `pthread_create()`로 동적 생성되며, 할당받은 단일 클라이언트와의 통신(`read/send`) 생명주기를 전담합니다.

### 메모리 관리와 데이터 안전성
1. **데이터 덮어쓰기(Data Race) 방지:** 메인 스레드의 소켓 변수가 덮어써지는 것을 막기 위해 접속 시마다 `malloc`으로 독립된 힙(Heap) 메모리를 생성하여 스레드에 안전하게 전달했습니다.
2. **독립된 수신 버퍼 (Stack 보장):** 각 스레드마다 독립된 스택(Stack) 메모리를 할당받아, 내부 수신 버퍼(`buffer`) 데이터가 클라이언트 간에 섞이지 않도록 처리했습니다.
3. **스레드 자원 자동 정리 (`pthread_detach`):** 통신이 끝난 스레드가 자신의 메모리(기본 8MB)를 즉시 커널에 반환하고 종료되도록 분리 상태로 생성하여 메모리 누수를 방지했습니다.

### 아키텍처의 한계 (Phase 3 도입 배경)
클라이언트가 늘어나는 수만큼 스레드도 동일하게 늘어나는 1:1 매핑 구조의 한계가 있습니다. 1만 명이 접속하는 **C10K 상황**에서는 누적되는 스레드 스택 메모리로 인해 서버 RAM이 고갈되며, 수많은 스레드를 교체하며 실행하는 **컨텍스트 스위칭(Context Switching)** 과정에서 CPU 오버헤드가 극심해져 서버가 다운됩니다.

## 🛠 Phase 3: 멀티플렉싱 서버 (epoll) - 현재 상태

### 구조 변경 내용
Phase 2의 메모리 및 CPU 과부하 문제를 해결하기 위해, 다수의 스레드를 폐기하고 **이벤트 기반의 넌블로킹 단일 스레드(Event-Driven Single Thread) 구조**로 전환했습니다. 

* `epoll_create1()`: 이벤트를 감지할 중앙 전광판(이벤트 큐)을 생성합니다.
* `epoll_ctl()`: 새로운 클라이언트 접속 시 해당 소켓에 알림 벨을 달아 큐에 등록합니다.
* `epoll_wait()`: 서버(단일 스레드)는 각 소켓에서 개별 대기하지 않고, 큐만 관찰하다가 실제 데이터가 도착한(이벤트가 발생한) 소켓 번호만 찾아가서 신속하게 처리합니다.

### 극대화된 메모리 효율과 안전성
1. **스레드 자원 경합 원천 차단:** 데이터를 처리하는 통로(스레드)가 하나뿐이므로, 멀티 스레드 환경에서 자원을 차지하기 위해 발생하는 교착상태(Deadlock)가 물리적으로 발생하지 않습니다.
2. **단일 공용 버퍼의 100% 재사용 (`memset` 활용):**
   스레드마다 수신 버퍼를 할당할 필요 없이, 단 1개의 공용 수신 버퍼(`buffer[1024]`)만 사용합니다. 새로운 데이터를 읽기 직전에 `memset()`으로 버퍼를 완전히 초기화하여, 이전 클라이언트의 찌꺼기 데이터가 섞이는 문제를 완벽히 방지하며 극강의 메모리 효율을 달성했습니다.

## 🚀 아키텍처의 한계 분석 및 향후 발전 방향

현재 구현된 Phase 3 서버는 메모리와 컨텍스트 스위칭 효율이 극대화되었으나, **단일 통로의 병목(Event Loop Blocking)**이라는 태생적 한계가 존재합니다. 특정 클라이언트가 CPU 연산이 오래 걸리는 무거운 작업(CPU-Bound Task)을 요청할 경우, 해당 처리가 끝날 때까지 이벤트 큐에 있는 나머지 수많은 클라이언트들의 통신이 모두 대기 상태에 빠지게 됩니다.

이를 해결하기 위해 향후 본 프로젝트는 Nginx나 Node.js(PM2) 등 글로벌 상용 서버들이 사용하는 **'멀티 워커(Multi-Worker) epoll 아키텍처'**로 확장할 계획입니다.

* **실무 표준 해결책 (CPU 코어 매칭):**
  무작정 스레드를 1만 개씩 늘리는 Phase 2의 방식으로 회귀하는 것이 아니라, 서버 컴퓨터의 **실제 CPU 코어(Core) 개수**에 정확히 맞춰서 epoll 워커 프로세스를 제한적으로 생성합니다.
* **기대 효과:**
  스레드가 코어 개수를 초과하지 않아 컨텍스트 스위칭 비용이 여전히 최소화되며, 1개의 워커가 무거운 작업으로 인해 병목에 빠지더라도 나머지 워커들이 남은 CPU 코어에서 트래픽을 병렬로 분산 처리하므로 전체 시스템의 다운을 방지할 수 있습니다.
