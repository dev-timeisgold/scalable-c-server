# C 네트워크 서버 프로젝트 (기본부터 대용량 처리까지)

C 언어의 소켓 프로그래밍을 이용해 기본 TCP 서버를 만들고, 구조적인 문제점을 찾아내며 점진적으로 성능을 개선해 나가는 프로젝트입니다.

## 진행 단계
- [x] Phase 1: 기본 TCP 에코 서버 (단일 스레드 / Blocking I/O)
- [x] Phase 2: 멀티 스레드 서버 (접속자당 1개의 스레드 할당)
- [ ] Phase 3: 멀티플렉싱 서버 (epoll을 이용한 대용량 처리)

---

## Phase 1: 기본 TCP 에코 서버 (단일 스레드)

### 구현 내용
리눅스 네트워크 기본 함수들을 순서대로 사용하여 통신 흐름을 만들었습니다.
1. `socket()`: TCP 통신을 위한 소켓(FD) 생성
2. `setsockopt()`: 서버 재시작 시 포트가 잠기는 에러 방지 (`SO_REUSEADDR`)
3. `bind()`: 소켓에 포트 번호(8080) 지정
4. `listen()`: 클라이언트 접속 대기열 생성
5. `accept()`: 접속한 클라이언트와 통신할 새로운 소켓 생성
6. `read()` / `send()`: 클라이언트의 메시지를 읽고 그대로 다시 전송(Echo)
7. `close()`: 통신 종료 후 자원 반환

### 문제점 (Phase 2로 넘어간 이유)
단일 스레드 구조이기 때문에 발생하는 **블로킹(Blocking) 문제**가 있었습니다. 
한 클라이언트가 접속한 후 메시지를 보내지 않고 가만히 있으면, 서버가 `read()` 함수에서 멈춰버립니다. 이 때문에 대기열에 있는 다음 클라이언트가 서버에 아예 들어오지 못하는 치명적인 문제가 발생했습니다.

## Phase 2: 멀티 스레드 서버 (현재 상태)

### 구조 변경 내용
Phase 1의 멈춤 현상을 해결하기 위해, 메인 스레드와 워커 스레드의 역할을 분리했습니다.
* **메인 스레드:** `accept()`를 통해 클라이언트의 접속만 받고 다시 대기합니다.
* **워커 스레드:** `pthread_create()`로 생성되며, 할당받은 클라이언트와 통신(`read/send`)을 전담합니다.

### 메모리 관리와 에러 방지
멀티 스레드 환경에서 발생할 수 있는 메모리 문제를 막기 위해 다음과 같이 코드를 구성했습니다.

1. **데이터 덮어쓰기(Data Race) 방지 (Heap 활용)**
   메인 스레드가 접속을 받을 때 사용하는 소켓 변수 하나를 그대로 스레드에 넘겨주면, 다음 접속자가 왔을 때 변수 값이 덮어써지는 문제가 생깁니다. 이를 막기 위해 접속이 올 때마다 `malloc`으로 독립된 힙(Heap) 메모리를 만들어서 소켓 번호를 안전하게 스레드에 넘겨주었습니다.

2. **독립된 버퍼 사용 (Stack 보장)**
   스레드가 생성될 때마다 운영체제가 각각 독립된 스택(Stack) 메모리를 할당해 줍니다. 따라서 스레드 내부에서 선언한 데이터 수신용 변수(`buffer`)는 여러 클라이언트가 동시에 접속해도 절대 서로 섞이지 않습니다.

3. **스레드 자원 정리 (`pthread_detach`)**
   서버의 특성상 메인 스레드가 워커 스레드의 종료를 기다려줄(`join`) 수 없습니다. 그래서 스레드를 생성할 때 `detach` 상태로 만들어, 통신이 끝나면 자신의 스택 메모리를 알아서 반환하고 종료되도록 처리했습니다.

### 문제점 (Phase 3로 넘어간 이유)
이 방식은 클라이언트가 늘어나는 만큼 스레드도 똑같이 늘어나는 한계가 있습니다. 
만약 1만 명이 접속하는 **C10K 상황**이 되면, 스레드 1만 개가 차지하는 메모리(RAM) 용량이 너무 커집니다. 또한, CPU가 수많은 스레드를 번갈아 가며 실행(Context Switching)하느라 실제 통신 처리보다 스레드 관리에 힘을 다 써버려 서버가 버티지 못하게 됩니다.
